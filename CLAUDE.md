# 먹점프 (Ink Jump) — 동양화 점프킹 프로젝트

> NHN NAN 2026 · Game x AI Hackathon 제출작
> Unity 기반 모바일 캐주얼 드로잉 클라이밍 게임 (Android APK 제출)

이 문서는 Claude Code와 Codex가 프로젝트 작업을 시작할 때 참고하는 최신 컨텍스트 파일이다.
현재 Unity 코어 루프, 드로잉 발판, 장애물, 아이템 5종, 먹분신, 사망 연출까지 구현되어
`main`에 병합된 상태다. 아래 11절의 현재 저장소 상태를 기준으로 후속 작업을 이어간다.

---

## 작업 시작 시 필수 절차

Claude Code를 포함한 모든 코딩 에이전트는 다음 순서를 지킨다.

1. 루트의 `AGENTS.md`와 이 `CLAUDE.md`를 끝까지 읽는다.
2. `git status --short --branch`로 현재 브랜치와 다른 작업자의 미커밋 변경을 확인한다.
3. `main`에서 직접 작업하지 않는다. 깨끗한 최신 `main`에서 `feature/*` 또는 `fix/*`
   브랜치를 만든 뒤 작업한다.
4. 구현에 앞서 관련 스크립트와 `MukJumpSceneBuilder.cs`의 현재 연결 구조를 확인한다.
5. 기능 구현 후 Unity 컴파일 로그와 변경 파일을 검증하고 AI 활용 기록을 갱신한다.

### 사용자와 합의된 기본 작업 흐름

- 기능 단위로 구현하고 Conventional Commit 형식의 작은 커밋을 남긴다.
- 완료된 커밋은 현재 기능 브랜치까지 `push`한다.
- `main` 병합은 사용자가 명시적으로 요청했을 때만 PR을 만들어 진행한다.
- 커밋 기록 보존을 위해 PR은 `Squash and merge`가 아니라 일반 `Merge pull request`를 쓴다.
- 병합 후 로컬 `main`을 원격과 동기화하고 병합된 기능 브랜치를 삭제한 뒤 다음
  `feature/*` 브랜치를 만든다.
- `git add -A`와 `git add .`은 사용하지 않고 이번 작업에서 수정한 파일만 명시적으로 추가한다.
- 다른 에이전트의 미커밋 변경은 커밋·리셋·체크아웃하지 않는다.
- GitHub로 push하거나 PR을 병합하는 외부 변경은 실행 전에 사용자 승인을 확인한다.

### Unity와 원격 조작 원칙

- `Assets/Scenes/Main.unity`를 YAML이나 Inspector에서 직접 편집하지 않는다.
  `Assets/Editor/MukJumpSceneBuilder.cs`를 수정하고 `MukJump > Build Main Scene`으로 재생성한다.
- Unity 에디터, Chrome, ChatGPT 등 GUI 원격 조작은 사용자가 “원격으로 해줘”라고
  명시했을 때만 수행한다. 그 외에는 코드와 에셋만 수정하고 필요한 메뉴 실행을 안내한다.
- Unity 에디터가 열려 있으면 같은 프로젝트를 `-batchmode`로 실행하지 않는다.
- 스크립트 수정 후 `~/Library/Logs/Unity/Editor.log`에서 `error CS`와 `Exception`을
  확인한 뒤 테스트를 요청한다.
- Input System 전용 프로젝트이므로 구형 `Input.*` 대신 `MukJump.Core.PointerInput`을 쓴다.
- 싱글톤은 Play 중 재컴파일을 고려해 `Instance`를 `OnEnable`에서 다시 할당한다.

### 문서 동기화 원칙

- 게임 규칙이나 구현 상태가 바뀌면 같은 작업에서 `README.md`, `docs/project-brief.md`,
  이 `CLAUDE.md` 중 영향을 받는 문서를 함께 갱신한다.
- AI를 사용한 코드·이미지·사운드·문서 작업은 즉시 `docs/ai-usage-log.md`에 기록한다.
- 외부 에셋을 추가하면 출처와 라이선스를 AI 활용 기록 상단 표에도 추가한다.

## 2. 게임 한 줄 컨셉

> "선 하나가 발판이 되고, 발판 하나가 그림이 된다"

동양화(수묵화) 붓질로 그리는 발판 + 자동으로 오르는 캐릭터 + AI가 완성하는 수묵화.
캐릭터는 조작할 수 없고 일정 주기마다 스스로 점프한다. 플레이어는 점프 타이밍에 맞춰
발판을 그려서 캐릭터의 다음 궤적(방향·높이)을 유도한다. 기존 '점프킹'이 점프력을
직접 조작하는 게임이라면, 이 게임은 발판을 그리는 행위 자체가 조작이 된다.

## 3. 게임 개요

| 항목 | 내용 |
|---|---|
| 장르 | 캐주얼 아케이드 클라이밍 (드로잉 플랫포머) |
| 플랫폼 | 모바일 — 이번 제출은 **Android APK** 기준, 세로 화면 |
| 엔진 | Unity (6000.3.10f1, Apple Silicon, LTS) / URP, Universal 2D 템플릿 |
| 타겟 | 전 연령, 캐주얼 퍼즐·아케이드 유저 |
| 플레이 타임 | 1회 1~3분, 반복 도전형 (하이스코어 경쟁) |
| 코어 루프 | 관찰 → 발판 스케치 → 자동 점프 → 상승 / 추락 → 재도전 |

## 4. 핵심 메커닉

- 캐릭터(먹방울이)는 **일정 주기마다 자동으로 점프**한다. 플레이어가 점프 자체를 조작할 수 없다.
- 플레이어는 화면에 **손가락으로 선을 그어 발판을 생성**한다.
- 발판의 **위치 · 각도 · 길이**가 다음 점프의 궤적(방향, 높이)을 결정한다.
- 그린 발판은 가파른 대각선에서도 캐릭터가 붙을 수 있다. 화면 양옆 `ScreenSideWall`은
  접착 대상이 아니며 기존처럼 안쪽으로 튕긴다.
- 첫 자동 점프는 접지 중 충전하고, 이후 충전은 점프 정점부터 시작한다. 발판 수명은
  기본 6.5초이며 동시에 최대 4개를 유지한다.
- 발판 붓결은 Main UI의 `LineSprite`를 우선 사용하고 없으면 절차적 갈필로 폴백한다.
- 발판을 잘못 그리거나 타이밍을 놓치면 캐릭터가 굴러 떨어진다 (기존 점프킹과 동일한 낙하 리스크).
- 전략의 핵심은 "내 점프 컨트롤"이 아니라 **"그림으로 만드는 길"**.

## 5. 캐릭터 — 먹방울이

센과 치히로의 숯검댕이(스스와타리)를 동양화 먹빛으로 재해석한 캐릭터.

**비주얼 스펙 (확정, `char_muk.svg` 참고 — 이미지 파일은 별도로 전달됨):**
- 몸통: 먹 번짐 텍스처의 불규칙한 스파이크형 검은 원 (붓 터치로 그린 잉크 블롯 실루엣)
- 눈: 크고 둥근 흰색 타원 + 검은 원형 동공. **눈썹 · 입 · 안광(하이라이트) 없음** — 표정은 눈 모양과 다리 포즈로만 표현
- 다리: 팔 없이 **다리만 존재**. 기본(점프 대기) 포즈는 무릎이 몸통 옆으로 뾰족하게
  튀어나온 산(山) 모양 실루엣 — 몸통 아래에서 위로 짧게 붙었다가 몸통 바깥쪽 높은
  지점(무릎)까지 솟은 뒤, 다시 아래 발까지 길게 떨어지는 꺾인 선. 발끝에는 작은 훅 모양 표시.
  → 웅크려서 점프를 준비하는 자세를 표현
- **그림자 없음** (땅에 닿는 그림자 연출은 추후 필요 시 게임 내 조명/이펙트로 별도 처리)
- 수집 오브젝트: 먹물방울 · 황금 붓 · 먹 방어막 · 먹분신의 4종 아이템을 사용한다.

**애니메이션 상태 (구현 완료):**
- 4×2 스프라이트시트 8프레임: idle · crouch · launch · rise · apex · fall · dive · land
- 사망 시 캐릭터가 먹 번짐으로 바뀌며, 해당 위치에 한지 먹 자국이 남는다.
- 먹분신 획득마다 한 마리씩 계속 누적되며, 한 마리가 살아 있으면 게임을 계속한다.

### 아이템 5종 확정 사양

| 아이템 | 효과 |
|---|---|
| 먹물방울 | 현재 위치에서 50m 상승. 상승 구간에는 장애물 피해를 받지 않음 |
| 황금 붓 | 8초 동안 먹 자원을 소모하지 않음 |
| 먹 방어막 | 장애물 피해 또는 화면 하단 추락을 1회 막고 다시 상승 |
| 먹분신 | 획득할 때마다 캐릭터 1마리 추가. 마지막 캐릭터가 죽을 때만 게임오버 |
| 붓 여유 | 기본 붓 용량의 35%를 100% 위에 누적. 그릴 때 먼저 소모되며 자연 회복되지 않음 |

- 아이콘 4종은 투명 1254×1254 PNG이며 실제 피사체 폭을 900px로 통일한다.
- 캐릭터와 같은 단순한 수묵 스타일, 제한된 검정·종이색·금색 팔레트, 굵기가 비슷한
  검정 외곽선을 유지한다.
- 먹분신은 넓고 납작한 먹 캐릭터 두 마리와 동일한 흰 타원 눈·검정 동공으로 표현한다.

## 6. AI 핵심 기능

### 6.1 실시간 수묵 변환 (해커톤 데모의 핵심 차별점)

플레이어가 대충 그린 발판 스케치(터치 스트로크)를 AI가 실시간으로 수묵화 붓질 질감의
발판 이미지로 자동 완성한다. 그림 실력과 무관하게 화면 전체가 하나의 산수화처럼
통일되는 것이 핵심 경험.

파이프라인: **스케치 입력 → AI 수묵 변환 → 발판 완성(인게임 콜라이더 반영)**

### 6.2 확장 아이디어 (시간 남으면)

- 절차적 산수화 배경: 고도에 따라 봄 매화 → 여름 폭포 → 가을 단풍 → 겨울 설산으로 배경 화풍 자동 전환
- 적응형 국악 사운드: 속도 · 위험도에 따라 국악기 기반 배경음을 AI가 실시간 변주
- 나만의 낙관(도장): 플레이 스타일을 분석해 엔딩에서 개인화된 도장 디자인을 AI가 생성

## 7. 기술 스택 & 구현 전략

- **엔진**: Unity 6 (6000.3.10f1), URP, Universal 2D 템플릿, 모바일(Android) 빌드 타겟
- **소스 관리**: Unity Editor Version Control Mode = Git (Meta 파일 보이기 + 텍스트 직렬화 설정됨)
- **드로잉 처리**: 터치 스트로크 캡처 → 베지어 스무딩 → 물리 콜라이더 자동 생성
- **AI 변환**: img2img 계열(스크리블 조건, 예: ControlNet-scribble) 스타일 고정 모델을
  우선 적용. 비동기 처리 + 캐시 프리셋 병행으로 체감 지연 최소화
- **폴백(Fallback)**: API 지연 · 실패 시 사전 정의된 먹 텍스처 셰이더로 즉시 대체 →
  끊김 없는 플레이 보장 (해커톤 데모에서 네트워크 이슈 대비 필수, "심사자가 별도 유료
  라이선스 없이 실행 가능"해야 하는 제출 요건과도 직결됨 — API 키 없이도 폴백만으로
  게임이 정상 진행되어야 함)

## 12. 리포 운영 방침 (커밋 기록이 심사 대상 — "정형화된 개발 프로세스" 흔적을 남길 것)

- 브랜치 전략: `main`(항상 빌드 가능) + `feature/*` 브랜치 → PR 머지. 2인 팀이므로 develop
  브랜치는 생략하고 feature → main PR로 단순화.
  - 예: `feature/auto-jump`, `feature/stroke-drawing`, `feature/ai-ink-pipeline`, `feature/android-build`
- 커밋 컨벤션: Conventional Commits — `feat:` `fix:` `chore:` `docs:` `art:` `build:` `refactor:`
  - 예: `feat(drawing): 터치 스트로크 캡처 및 베지어 스무딩 추가`
- 작업 단위: 기능 하나 = 브랜치 하나 = PR 하나. 커밋은 의미 단위로 잘게 나눠서, 한 번에
  몰아서 커밋하지 않는다 (심사자가 커밋 히스토리를 본다는 전제).
- 대용량 바이너리(아트 원본 등)가 커지면 Git LFS 검토.
- README에 실행 방법·빌드 방법을 반드시 명시 (심사자 실행 가능성 요건, 3절 참고).
- 현재 합의된 한 사이클:
  `feature/* 작업 → 기능별 commit → push → 사용자 요청 시 PR 일반 merge → main pull →
  병합 브랜치 삭제 → 다음 feature 브랜치`.

## 13. AI 활용 내역 기록 지침 (4번 제출물의 원본 자료)

**`docs/ai-usage-log.md` 파일을 만들고, 개발 중 AI 도구를 쓸 때마다 그때그때 기록할 것.**

```
### YYYY-MM-DD — [작업 항목]
- 사용 도구: (예: Claude Code, Stable Diffusion + ControlNet 등)
- 목적: (예: AutoJump 스크립트 작성 / 발판 수묵 변환 프롬프트 설계)
- 주요 프롬프트/지시: (실제 사용한 프롬프트 원문 또는 핵심 지시사항 요약)

- 사람의 수정/검토 내용
```

외부 에셋/오픈소스 사용 시 출처·라이선스도 같은 파일 상단에 표로 정리.
