## 2. 게임 한 줄 컨셉

> "선 하나가 발판이 되고, 발판 하나가 그림이 된다"

동양화(수묵화) 붓질로 그리는 발판 + 자동으로 오르는 캐릭터 + AI가 완성하는 수묵화.
캐릭터는 조작할 수 없고 일정 주기마다 스스로 점프한다. 플레이어는 점프 타이밍에 맞춰
발판을 그려서 캐릭터의 다음 궤적(방향·높이)을 유도한다. 기존 '점프킹'이 점프력을
직접 조작하는 게임이라면, 이 게임은 발판을 그리는 행위 자체가 조작이 된다.

## 3. 게임 개요

| 항목 | 내용 |
|---|---|
| 장르 | 캐주얼 아케이드 클라이밍 (드로잉 플랫포머) |
| 플랫폼 | 모바일 — 이번 제출은 **Android APK** 기준, 세로 화면 |
| 엔진 | Unity (6000.3.10f1, Apple Silicon, LTS) / URP, Universal 2D 템플릿 |
| 타겟 | 전 연령, 캐주얼 퍼즐·아케이드 유저 |
| 플레이 타임 | 1회 1~3분, 반복 도전형 (하이스코어 경쟁) |
| 코어 루프 | 관찰 → 발판 스케치 → 자동 점프 → 상승 / 추락 → 재도전 |

## 4. 핵심 메커닉

- 캐릭터(먹방울이)는 **일정 주기마다 자동으로 점프**한다. 플레이어가 점프 자체를 조작할 수 없다.
- 플레이어는 화면에 **손가락으로 선을 그어 발판을 생성**한다.
- 발판의 **위치 · 각도 · 길이**가 다음 점프의 궤적(방향, 높이)을 결정한다.
- 발판을 잘못 그리거나 타이밍을 놓치면 캐릭터가 굴러 떨어진다 (기존 점프킹과 동일한 낙하 리스크).
- 전략의 핵심은 "내 점프 컨트롤"이 아니라 **"그림으로 만드는 길"**.

## 5. 캐릭터 — 먹방울이

센과 치히로의 숯검댕이(스스와타리)를 동양화 먹빛으로 재해석한 캐릭터.

**비주얼 스펙 (확정, `char_muk.svg` 참고 — 이미지 파일은 별도로 전달됨):**
- 몸통: 먹 번짐 텍스처의 불규칙한 스파이크형 검은 원 (붓 터치로 그린 잉크 블롯 실루엣)
- 눈: 크고 둥근 흰색 타원 + 검은 원형 동공. **눈썹 · 입 · 안광(하이라이트) 없음** — 표정은 눈 모양과 다리 포즈로만 표현
- 다리: 팔 없이 **다리만 존재**. 기본(점프 대기) 포즈는 무릎이 몸통 옆으로 뾰족하게
  튀어나온 산(山) 모양 실루엣 — 몸통 아래에서 위로 짧게 붙었다가 몸통 바깥쪽 높은
  지점(무릎)까지 솟은 뒤, 다시 아래 발까지 길게 떨어지는 꺾인 선. 발끝에는 작은 훅 모양 표시.
  → 웅크려서 점프를 준비하는 자세를 표현
- **그림자 없음** (땅에 닿는 그림자 연출은 추후 필요 시 게임 내 조명/이펙트로 별도 처리)
- 수집 오브젝트(향후 구현): 별사탕 대신 매화 · 단풍잎 · 도장 조각 등 동양화 소재

**애니메이션 상태 (구현 필요, 아직 아트 없음):**
- Idle / 점프 대기 (완료된 기본 포즈)
- 점프 중 (다리를 뒤로 뻗는 등 변형 필요)
- 착지
- 낙하 / 실패

## 6. AI 핵심 기능

### 6.1 실시간 수묵 변환 (해커톤 데모의 핵심 차별점)

플레이어가 대충 그린 발판 스케치(터치 스트로크)를 AI가 실시간으로 수묵화 붓질 질감의
발판 이미지로 자동 완성한다. 그림 실력과 무관하게 화면 전체가 하나의 산수화처럼
통일되는 것이 핵심 경험.

파이프라인: **스케치 입력 → AI 수묵 변환 → 발판 완성(인게임 콜라이더 반영)**

### 6.2 확장 아이디어 (시간 남으면)

- 절차적 산수화 배경: 고도에 따라 봄 매화 → 여름 폭포 → 가을 단풍 → 겨울 설산으로 배경 화풍 자동 전환
- 적응형 국악 사운드: 속도 · 위험도에 따라 국악기 기반 배경음을 AI가 실시간 변주
- 나만의 낙관(도장): 플레이 스타일을 분석해 엔딩에서 개인화된 도장 디자인을 AI가 생성

## 7. 기술 스택 & 구현 전략

- **엔진**: Unity 6 (6000.3.10f1), URP, Universal 2D 템플릿, 모바일(Android) 빌드 타겟
- **소스 관리**: Unity Editor Version Control Mode = Git (Meta 파일 보이기 + 텍스트 직렬화 설정됨)
- **드로잉 처리**: 터치 스트로크 캡처 → 베지어 스무딩 → 물리 콜라이더 자동 생성
- **AI 변환**: img2img 계열(스크리블 조건, 예: ControlNet-scribble) 스타일 고정 모델을
  우선 적용. 비동기 처리 + 캐시 프리셋 병행으로 체감 지연 최소화
- **폴백(Fallback)**: API 지연 · 실패 시 사전 정의된 먹 텍스처 셰이더로 즉시 대체 →
  끊김 없는 플레이 보장 (해커톤 데모에서 네트워크 이슈 대비 필수, "심사자가 별도 유료
  라이선스 없이 실행 가능"해야 하는 제출 요건과도 직결됨 — API 키 없이도 폴백만으로
  게임이 정상 진행되어야 함)

## 12. 리포 운영 방침 (커밋 기록이 심사 대상 — "정형화된 개발 프로세스" 흔적을 남길 것)

- 브랜치 전략: `main`(항상 빌드 가능) + `feature/*` 브랜치 → PR 머지. 2인 팀이므로 develop
  브랜치는 생략하고 feature → main PR로 단순화.
  - 예: `feature/auto-jump`, `feature/stroke-drawing`, `feature/ai-ink-pipeline`, `feature/android-build`
- 커밋 컨벤션: Conventional Commits — `feat:` `fix:` `chore:` `docs:` `art:` `build:` `refactor:`
  - 예: `feat(drawing): 터치 스트로크 캡처 및 베지어 스무딩 추가`
- 작업 단위: 기능 하나 = 브랜치 하나 = PR 하나. 커밋은 의미 단위로 잘게 나눠서, 한 번에
  몰아서 커밋하지 않는다 (심사자가 커밋 히스토리를 본다는 전제).
- 대용량 바이너리(아트 원본 등)가 커지면 Git LFS 검토.
- README에 실행 방법·빌드 방법을 반드시 명시 (심사자 실행 가능성 요건, 3절 참고).

## 13. AI 활용 내역 기록 지침 (4번 제출물의 원본 자료)

**`docs/ai-usage-log.md` 파일을 만들고, 개발 중 AI 도구를 쓸 때마다 그때그때 기록할 것.**
제출 직전에 몰아서 복원하려 하면 반드시 누락된다. 기록 형식:

```
### YYYY-MM-DD — [작업 항목]
- 사용 도구: (예: Claude Code, Stable Diffusion + ControlNet 등)
- 목적: (예: AutoJump 스크립트 작성 / 발판 수묵 변환 프롬프트 설계)
- 주요 프롬프트/지시: (실제 사용한 프롬프트 원문 또는 핵심 지시사항 요약)
- 결과물: (커밋 링크 또는 파일 경로)
- 사람의 수정/검토 내용
```

외부 에셋/오픈소스 사용 시 출처·라이선스도 같은 파일 상단에 표로 정리.
